# 2026-07-02 | Repository management
- - -
Стратегии именования веток в проекте:
- **feature/** - новая функциональность, улучшения кода
- **fix/ | hotfix/** - исправления ошибок, багов
- **refactor/** - рефакторинг кода
- **docs/** - написание документации

Стратегии именования коммитов:
- **feat** - новая функциональность, улучшение
- **fix** - исправление ошибки
- **refactor** - рефакторинг кода (без изменения функционала)
- **docs** - только документация
- **test** - добавлены или изменены тесты
- **build** - изменения в сборке проекта
- **chore** - всё остальное(dependencies, gradle, etc)
- **style** - изменение стиля кода
- **revert** - отмена коммита

Структура коммита:
```text
# <тип>(<область>): <краткое описание>
#
# Подробное описание (если нужно)
# - Что было сделано
# - Зачем это сделано
# - Какую проблему решает
#
# Ссылки:
# - Closes #123
# - Fixes #456
```

# 2026-07-03 | Squash commits naming
- - -
Если ты используешь стратегию squash коммитов, при которой все коммиты твоей ветки схлопываются в один и перебазируются на **master**, то нужно давать понятные названия данным коммитам, чтобы другие люди понимали, что ты вообще в этом коммите сделал.

Можно воспользоваться следующим способом:

```
<type>(<scope>): <short_description>
```

Где:

- `<type>` - feat, fix, refactor, docs, etc
- `<scope>` - db, auth, websocket, api, etc

А в описании коммита у тебя в идеале должно быть описание всех коммитов из твоей ветки. Обычно это автоматически делает **GitHub/GitLab**
